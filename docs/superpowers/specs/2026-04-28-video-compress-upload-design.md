# 超过 100MB 的视频下载后压缩再上传设计

## 目标
当链接视频下载到本地后，如果实际文件大小超过 100MB，则先进行本地压缩转码，再把压缩后的临时文件交给后续生成流程。

## 背景
当前生成链路会在下载阶段按 `content-length` / 实际下载字节数拦截超大视频。对于某些 Douyin 链接，TikHub 返回的高画质播放地址可能超过 100MB，但用户仍希望尽量完成生成，而不是直接失败。

## 范围
- 仅影响链接模式下的视频下载链路
- 仅对下载后实际大小超过 100MB 的视频触发压缩
- 小于等于 100MB 的视频保持原样，不额外转码
- 压缩后的文件仍以临时文件形式进入现有生成流程
- 暂不改动上传模式的文件处理逻辑

## 推荐方案
### 1. 下载后阈值压缩
1. 先下载视频到本地临时文件
2. 读取实际文件大小
3. 若 `<= 100MB`，直接返回原文件
4. 若 `> 100MB`，调用本地 `ffmpeg` 进行压缩转码
5. 输出一个新的临时 MP4 文件，再交给后续生成流程

这是推荐方案，因为它只在必要时增加额外处理，且不依赖 TikHub 必须返回某个特定清晰度链接。

## 压缩策略
- 输出格式统一为 MP4
- 目标优先级：先保证体积下降，再保证内容可用
- 采用相对保守但稳定的转码参数，倾向于：
  - 降低分辨率到 720p 级别
  - 降低视频码率
  - 音频使用较低码率 AAC
- 如果第一次压缩后仍然超过 100MB，可再做一次更强压缩；若仍失败，则返回明确错误

## 预期实现位置
### 服务层
建议在 `server/services/file.ts` 中新增视频压缩能力，和当前下载、落盘、清理逻辑保持同层。

可拆成以下职责：
- `getFileSize(path)`：读取临时文件实际大小
- `compressVideoFile(sourcePath, targetOptions)`：基于 `ffmpeg` 输出压缩文件
- `ensureVideoUnderLimit(path, maxBytes)`：判断是否需要压缩并返回最终可用文件
- `downloadVideoUrlToTempFile(...)` 需要支持“可压缩下载”路径：当调用方希望在超限后压缩时，下载阶段不能提前按 100MB 截断，否则根本拿不到可压缩的源文件

### 调用层
在 `server/api/generate.post.ts` 的链接下载路径中：
1. 下载完成后拿到临时文件路径
2. 传入 `ensureVideoUnderLimit(...)`
3. 后续生成流程只接收最终文件路径

## 数据流
1. 用户提交 `link` 模式请求
2. 解析 Douyin 链接，获取播放 URL
3. 下载视频到本地临时目录
4. 检查文件大小
5. 超过 100MB 时压缩到新的临时文件
6. 将最终文件路径交给 DashScope 生成
7. 结束后清理原始文件和压缩文件

## 错误处理
- 下载失败：保持现有 `VIDEO_FETCH_FAILED`
- 进入压缩流程的下载路径应允许下载到超过 100MB 的源文件，然后再交由压缩器处理；不要在下载阶段先把它当成致命超限
- 压缩工具不可用：返回明确的压缩失败错误
- 压缩超时：返回可重试错误
- 压缩后仍超 100MB：返回明确的超限错误，提示用户重试或换更小来源
- 清理失败：记录日志，不阻塞主流程返回

## 兼容性
- 不影响小视频的现有路径
- 不影响 upload 模式
- 不改变生成结果的接口格式
- 不改变前端表单结构

## 测试计划
### 单元测试
- 小于等于 100MB 的文件不会触发压缩
- 大于 100MB 的文件会触发压缩
- 压缩成功后返回新文件路径
- 压缩失败时会抛出可识别错误

### API 测试
- link 模式大文件会走压缩再生成
- link 模式小文件不走压缩
- 压缩后的文件会继续进入模型生成调用

## 风险与权衡
- 压缩会增加额外耗时，尤其是大视频
- 转码质量会下降，但这是用户接受的取舍
- `ffmpeg` 依赖系统环境，部署镜像需要保证可用
- 如果未来需要更高吞吐量，可能需要把压缩任务外置成独立 worker

## 结论
这是一个“下载后再压缩”的阈值策略，能尽量保住可生成性，同时只在超过 100MB 时付出额外成本。
