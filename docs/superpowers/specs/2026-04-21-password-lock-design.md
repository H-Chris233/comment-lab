# 密码锁页面设计

## 目标
为整个站点增加一个单密码锁：首次访问先设置密码；之后访问需输入密码解锁；进入主页面后可随时修改密码。

## 范围
- 仅做一个全站统一密码
- 密码对所有设备/浏览器生效
- 解锁状态仅保持到浏览器会话结束
- 刷新页面后仍保持解锁
- 主页面内提供修改密码入口

## 状态机
### 1. 未设置密码
- 服务端还没有密码记录
- 页面显示“设置密码”表单
- 设置成功后自动进入已解锁状态

### 2. 已设置密码，未解锁
- 页面显示“输入密码”表单
- 密码正确后发放会话 Cookie
- 失败时提示密码错误

### 3. 已解锁
- 显示原有评论生成主页面
- 主页面提供“修改密码”入口
- 修改密码后当前会话继续有效

## 认证策略
- 服务端保存密码哈希，不保存明文
- 服务端发放 HTTP-only 会话 Cookie
- Cookie 仅用于当前浏览器会话
- 页面刷新后通过 Cookie 恢复解锁状态

## 存储策略
- 密码状态保存在服务端本地文件中
- 文件放在共享的 `.tmp` 目录下，适配 Docker Compose
- 文件内只保存：
  - 密码哈希
  - 盐值
  - 会话签名密钥
  - 时间戳

## API 设计
- `GET /api/auth/status`
  - 返回是否已设置密码、当前是否已解锁
- `POST /api/auth/set-password`
  - 首次设置密码
- `POST /api/auth/login`
  - 输入密码解锁
- `POST /api/auth/change-password`
  - 在已解锁状态下修改密码
- `POST /api/auth/logout`
  - 可选：清除当前会话 Cookie

## 访问控制
- `POST /api/generate` 和 `POST /api/parse-link` 必须在解锁后才能访问
- 锁定状态下返回 401/未授权错误

## 前端体验
- 首次访问时先展示锁页
- 锁页分成两种文案：
  - 首次设置密码
  - 输入密码进入
- 解锁后显示主页面
- 主页面顶部或设置区提供修改密码

## 风险与兜底
- 密码状态文件缺失时视为“未设置密码”
- 会话 Cookie 仅由浏览器会话生命周期管理
- 修改密码不应踢掉当前会话
